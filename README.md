# Admin — переиспользуемый каркас админ-панели

© 2026 Oleksandr Nosov. Лицензия — [MIT](LICENSE).

Референсный проект админ-панели: список с фильтром и пейджингом, универсальное окно
детальной информации (плавающее / докер справа / докер снизу), JSON-driven конфигурация
списков (без копипасты одной и той же таблицы на каждой странице), inline-редактирование
прямо в таблице. Frontend — Vue 3 + Vite. Backend — Yii3 + SQLite, наполняется из CSV
и полностью сбрасывается при каждом рестарте контейнера (это фейковый бэкенд для демонстрации
и разработки, а не для хранения настоящих данных).

Живое демо: https://admin.4n.com.ua (логин `admin` / пароль `admin123`).

## 1. Как развернуть проект отдельно, чтобы посмотреть

Нужен только Docker Desktop (или Docker + Docker Compose на Linux).

```bash
git clone https://github.com/alex-no/admin.git
cd admin
docker compose up -d --build
```

Первый запуск займёт несколько минут — фронтенд ставит npm-зависимости, бэкенд разворачивает
Yii3 через `composer create-project` и сеет SQLite из CSV. Дальше запуски быстрые.

После старта:
- Фронтенд (dev-сервер с hot-reload): **http://localhost:5173**
- Бэкенд напрямую (для отладки API): **http://localhost:8080**
- Swagger-документация API: **http://localhost:5173/api/admin/doc**
- Логин: `admin` / `admin123` (суперадмин) или `manager` / `manager123` (ограниченные права) —
  см. `backend/data/csv/users.csv`

Данные (СТО, країни, користувачі) лежат в `backend/data/csv/*.csv` — правите CSV,
перезапускаете бэкенд (`docker compose restart backend`) — новые данные подхватываются
при следующем старте контейнера.

Кроме основной демки списка (**Дані → Реєстр даних**, `/data-registry` — витрина каркасу
`list-framework`, описана ниже), в проекте есть ещё один блок демо-страниц, показывающий, что
тот же подход (фейковый SQLite-бэкенд, данные сбрасываются при рестарте) масштабируется и
на более сложные, не-табличные разделы админки — **Аналітика** (`/analytics`, `/analytics/stats`,
`/analytics/charts`) и **Логи помилок** (`/error-logs`, `/error-logs/stats`). Данные для них
генерируются процедурно при каждом старте контейнера (`backend/data/fake_analytics.php`), а не
берутся из CSV — там от природы большой случайный time-series набор. Эти страницы написаны
не через `list-framework` (перенесены как есть из продакшн-проекта AllSTO), поэтому как образец
для раздела 4 ориентируйтесь на `StoRegistry.vue`, а не на них. У детальной карточки визита
в Аналітиці есть вкладка "Діагностика" (ping/traceroute/reverse-DNS/blacklist-check/HTTP-заголовки/
бан IP) — это **настоящие** сетевые операции (`backend/src/Admin/Controller/AdminNetworkToolsController.php`),
а не заглушки; при переносе в другой проект их стоит либо оставить как есть (админ-only,
IP валидируется перед подстановкой в shell), либо вовсе не переносить, если такой функционал не нужен.

### Важно про live-reload

На некоторых конфигурациях Windows + Docker Desktop (особенно не системный диск) file-watcher
Vite не видит правки файлов на лету. Если после правки кода браузер не обновился — сделайте
`docker compose restart frontend` (для фронта) или `docker compose restart backend` (для бэка,
PHP перечитывается на каждый запрос, но правки `composer.json`/добавление нового файла в новый
неймспейс иногда тоже требуют рестарта).

### Продакшн-вариант

`docker-compose.prod.yml` — единый контейнер (Apache+PHP), который отдаёт и собранный
статический фронтенд (`npm run build`), и API из одного и того же процесса. Именно так
сейчас развёрнуто на `admin.4n.com.ua`. Смотрите его как пример деплоя за внешним
reverse-proxy (в проде — хостовый Apache с Let's Encrypt).

## 2. Как интегрировать эту админку в другой проект

Основная идея проекта: большая часть кода админки — **не про конкретный продукт**, а про
переиспользуемые механизмы. В любой новый проект переносятся именно они, а бизнес-специфика
(какие сущности, какие поля) описывается конфигом.

### Что переиспользуется целиком

| Механизм | Где лежит | Что делает |
|---|---|---|
| Универсальное окно деталей | `frontend/src/components/BaseModal.vue` + `frontend/src/composables/useModalWindow.js` | Три режима (floating / docked-right / docked-bottom), drag & resize, автосохранение режима в localStorage. Контент — через slots (`#title`, default, `#footer`) |
| Каркас списка (список+фильтр+пейджинг) | `frontend/src/list-framework/` | `DataListPage.vue` + реестры типов (`filterTypes.js`, `cellTypes.js`) + `useRemoteOptions.js`. Колонки и фильтры задаются JSON-конфигом, а не хардкодятся в шаблоне |
| Сдвиг контента при открытом докере | `frontend/src/components/ListPageWrapper.vue` + `usePageLayout.js` | Докнутое окно не перекрывает список, а сдвигает его |
| Синхронизация фильтров/сортировки/открытой карточки с URL | `frontend/src/composables/useUrlFilters.js` | Фильтры, сортировка (в т.ч. мульти-колоночная) и открытая детальная карточка (id + активная вкладка) переживают перезагрузку страницы и передаются в ссылке — см. 4.2 |
| Меню, управляемое JSON | `frontend/src/config/menu.json` | Пункты меню + права доступа + (задел) роли — без правки Vue-компонентов |
| Авторизация с dev-заглушкой | `frontend/src/composables/useAuth.js` | `VITE_MOCK_AUTH=true` — пускает с любыми данными, пока нет настоящего бэкенда; `=false` — ходит в реальный `/api/admin/auth/*` |

### Шаги интеграции в существующий проект

1. **Скопируйте** в целевой Vue 3 + Vite проект:
   - `frontend/src/list-framework/`
   - `frontend/src/components/BaseModal.vue`, `Pagination.vue`, `SortIcon.vue`, `ListPageWrapper.vue`
   - `frontend/src/composables/useModalWindow.js`, `usePageLayout.js`, `useUrlFilters.js`, `useAuth.js`
   - `frontend/src/config/menu.json` (как образец структуры, наполните своими пунктами)
2. **Подключите свой backend.** `DataListPage.vue` ожидает от `apiList` ответ вида
   `{ status, data: [...], pagination: { page, per_page, total, total_pages } }`,
   а `apiUpdate` — `PATCH /{id}` с телом `{ поле: значение }`. Если у вашего API другой
   формат — либо адаптируйте его под этот контракт, либо поправьте `load()`/`handleCellUpdate()`
   в `DataListPage.vue` под свой. Если хотите, чтобы ссылка на открытую детальную карточку
   (`?id=...`) работала при прямом переходе/перезагрузке (а не только при клике по строке
   уже загруженного списка) — добавьте также `GET {apiUpdate}/{id}` (один объект), см.
   `AdminStoController::show()` и `StoRegistry.vue`'s `useUrlFilters({ detail: { id, onOpen } })`
   как образец.
3. **Опишите свою сущность** тремя JSON-файлами по образцу
   `frontend/src/pages/sto-registry.{filter,columns,config}.json` (подробнее — раздел 4).
4. **Стили.** Сейчас используется Bootstrap 5 — если в целевом проекте другой UI-кит,
   компоненты `list-framework/filters/*` и `list-framework/cells/*` достаточно простые,
   чтобы переписать разметку под ваш дизайн-систем, не трогая логику в `DataListPage.vue`.

### Backend — по желанию

Бэкенд на Yii3 в этом проекте — **демонстрационный** (фейковые данные, SQLite вместо реальной
БД). Для интеграции в реальный проект backend обычно берётся не отсюда, а пишется под
реальную БД вашего проекта — важно лишь повторить контракт ответов (см. выше) и, если
нужен Swagger, — стиль аннотаций `#[OA\...]` из `backend/src/Admin/Controller/*`
(атрибуты пакета `zircote/swagger-php`).

## 3. Интеграция с помощью AI (Claude)

Этот проект специально спроектирован так, чтобы интеграцию делал не только человек руками,
а в первую очередь — AI-ассистент (например, [Claude Code](https://claude.com/claude-code)),
которому дали доступ и к этому репозиторию, и к целевому проекту. Вся механика вынесена
в понятные, документированные, небольшие файлы (JSON-конфиги, реестры типов) именно для того,
чтобы её было легко "прочитать" и повторить по аналогии — это ровно то, в чём AI-ассистенты
особенно хороши.

Практический сценарий:

1. Откройте Claude Code (или аналог) в целевом проекте, к которому нужно подключить админку.
2. Дайте ему доступ так же и к этому репозиторию (например, клонируйте рядом).
3. Попросите примерно так:

   > Изучи `frontend/src/list-framework` и `frontend/src/components/BaseModal.vue`
   > в проекте `admin` — это каркас списка с фильтром/пейджингом и универсального окна
   > деталей. Перенеси эти механизмы в мой проект и подключи их для сущности `<название>`:
   > колонки `<...>`, фильтры `<...>`, детальная карточка — докер справа. Смотри на
   > `frontend/src/pages/StoRegistry.vue` и `sto-registry.*.json` как на образец, как
   > это сочетается на одной странице.

4. Для нестандартных элементов (свой тип фильтра, своя ячейка таблицы) — попросите
   зарегистрировать новый тип через `registerFilterType()`/`registerCellType()`
   (см. раздел 4) вместо того, чтобы писать список руками с нуля.

AI хорошо справляется именно с этим видом работы: контракт (JSON-схема, форма ответа API)
чёткий и небольшой, а типовых страниц обычно много и они однообразны — то есть после
одной показанной вручную интеграции остальные сущности AI может добавлять по аналогии
почти самостоятельно.

## 4. Как добавлять новые страницы и элементы

### 4.1. Простая страница (без списка)

1. Создайте `frontend/src/pages/MyPage.vue` (обычный Vue-компонент, оберните в `BaseLayout`
   или `ListPageWrapper`, если планируете докер-окно).
2. Добавьте маршрут в `frontend/src/router/index.js`:
   ```js
   { path: '/my-page', name: 'MyPage', component: () => import('../pages/MyPage.vue') },
   ```
3. Добавьте пункт в `frontend/src/config/menu.json` (в существующую группу или новую):
   ```json
   { "label": "Моя сторінка", "to": "/my-page", "icon": "bi-star", "permission": "my.view", "roles": ["admin", "superadmin"] }
   ```

### 4.2. Список с фильтром и пейджингом (типовой случай)

Возьмите за образец `frontend/src/pages/sto-registry.*.json` и `StoRegistry.vue`.

1. **Колонки** — `my-entity.columns.json`, массив объектов:
   ```json
   { "key": "name", "label": "Назва", "type": "text", "sortable": true, "editable": true }
   ```
   `type` — берётся из реестра `frontend/src/list-framework/cellTypes.js`
   (`text`, `select`, `boolean`, `number`, `phone-list` — последний хранит массив строк и
   форматирует их под "+38 (0XX) XXX-XX-XX", см. `cells/PhoneListCell.vue` и `utils/phone.js`
   как готовый образец кастомного типа со своим форматированием и E.164-нормализацией при
   сохранении). `editable: true` включает inline-редактирование прямо в таблице (без
   `editable` — только чтение).

   `sortable: true` включает клик по заголовку колонки для сортировки. Обычный клик —
   сортировка только по этой колонке; **Ctrl/Cmd+клик** — добавляет колонку к уже выбранной
   сортировке (мульти-колоночная сортировка, порядок кликов = приоритет). Состояние хранится
   в URL одним параметром `?sort=key:ASC,key2:DESC` (через `multiSort` у `useUrlFilters`),
   на бэкенд передаётся как `sort_by=key,key2&sort_dir=ASC,DESC` (список полей/направлений
   в одном и том же порядке). **Важный нюанс:** если поле в БД хранит код/id, а пользователю
   показывается подпись (напр. `select`-колонка с `options`) — сортировать по сырому коду
   неправильно, нужно сортировать по слову, которое видно в таблице. На бэкенде для такого
   поля подменяйте `ORDER BY` на `CASE ... WHEN 'code' THEN 'Label' ... END` — пример:
   `AdminStoController::sortExpr()` для колонки `sto_type`.

2. **Фильтр** — `my-entity.filter.json`, аналогично, типы — из `filterTypes.js`
   (`text`, `select`, `checkbox`). Для выпадающего списка, наполняемого с бэкенда:
   ```json
   { "key": "country_id", "type": "select", "label": "Країна", "optionsUrl": "/api/admin/geography/countries", "optionsValueKey": "id", "optionsLabelKey": "name_uk" }
   ```
   Опции кэшируются по URL — один и тот же справочник не грузится повторно, даже если
   используется в нескольких полях/страницах.

3. **Конфиг эндпоинтов** — `my-entity.config.json`: `apiList`/`apiUpdate`/`apiDelete`,
   разрешения, кнопки действий в строке (`detail`, `delete`, кастомные).

4. **Страница** — тонкий `.vue`-файл, который просто передаёт эти три JSON в `DataListPage`
   (см. `StoRegistry.vue` целиком — там же пример открытия детальной карточки в `BaseModal`
   по клику на кнопку `detail`). Там же — образец, как сделать так, чтобы ссылка на открытую
   карточку (и, если у карточки несколько вкладок — на конкретную вкладку) переживала
   перезагрузку и годилась для передачи другому человеку: свой `useUrlFilters({ filters: { tab: activeTab }, detail: { id: detailId, onOpen } })`
   на уровне страницы (отдельно от того, что `DataListPage` уже делает для фильтров списка) —
   `onOpen(id)` подгружает запись через `GET {apiUpdate}/{id}` и открывает `BaseModal`,
   `activeTab` синхронизируется как обычный фильтр. В `sto-registry.config.json` кнопки
   действий (`actions`) тоже могут открывать сразу конкретную вкладку через `"tab": "..."`.

### 4.3. Свой (кастомный) элемент фильтра или колонки

Если библиотечных типов (`text`/`select`/`boolean`/`number`/`checkbox`) не хватает:

```js
// в вашем коде инициализации (например, main.js) или прямо в компоненте страницы
import { registerCellType } from '@/list-framework/cellTypes'
import MyRatingCell from '@/components/MyRatingCell.vue'

registerCellType('my-rating', MyRatingCell)
```

Компонент должен принимать props `field`, `modelValue`, `readonly`, `row` и эмитить
`update:modelValue` — тот же контракт, что и у встроенных `TextCell.vue`/`SelectCell.vue`
(смотрите любой из них как образец). После регистрации просто укажите `"type": "my-rating"`
в JSON-конфиге колонки.

### 4.4. Детальная карточка (окно)

Любое количество вкладок/полей — обычная разметка внутри `<BaseModal>`. Три режима
отображения, drag/resize, сохранение размеров — уже работают без дополнительного кода.
Смотрите `frontend/docs/BASE_MODAL_USAGE.md` и `frontend/docs/MODAL_WINDOWS.md` — там подробное описание
всех props и слотов с примерами.
