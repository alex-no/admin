# Admin — перевикористовуваний каркас адмін-панелі

© 2026 Oleksandr Nosov. Ліцензія — [MIT](LICENSE).

Референсний проект адмін-панелі: список із фільтром і пейджингом, універсальне вікно
детальної інформації (плаваюче / докер справа / докер знизу), JSON-driven конфігурація
списків (без копіпасти однієї й тієї ж таблиці на кожній сторінці), inline-редагування
прямо в таблиці. Frontend — Vue 3 + Vite. Backend — Yii3 + SQLite, наповнюється з CSV
і повністю скидається при кожному рестарті контейнера (це фейковий бекенд для демонстрації
та розробки, а не для зберігання справжніх даних).

Живе демо: https://admin.4n.com.ua (логін `admin` / пароль `admin123`).

## 1. Як розгорнути проект окремо, щоб подивитися

Потрібен лише Docker Desktop (або Docker + Docker Compose на Linux).

```bash
git clone https://github.com/alex-no/admin.git
cd admin
docker compose up -d --build
```

Перший запуск займе кілька хвилин — фронтенд ставить npm-залежності, бекенд розгортає
Yii3 через `composer create-project` і сіє SQLite з CSV. Далі запуски швидкі.

Після старту:
- Фронтенд (dev-сервер з hot-reload): **http://localhost:5173**
- Бекенд напряму (для відлагодження API): **http://localhost:8080**
- Swagger-документація API: **http://localhost:5173/api/admin/doc**
- Логін: `admin` / `admin123` (суперадмін) або `manager` / `manager123` (обмежені права) —
  див. `backend/data/csv/users.csv`

Дані (СТО, країни, користувачі) лежать у `backend/data/csv/*.csv` — правите CSV,
перезапускаєте бекенд (`docker compose restart backend`) — нові дані підхоплюються
при наступному старті контейнера.

Окрім основної демки списку (**Дані → Реєстр даних**, `/data-registry` — вітрина каркасу
`list-framework`, описана нижче), у проекті є ще один блок демо-сторінок, що показує, що
той самий підхід (фейковий SQLite-бекенд, дані скидаються при рестарті) масштабується і
на складніші, не-табличні розділи адмінки — **Аналітика** (`/analytics`, `/analytics/stats`,
`/analytics/charts`) і **Логи помилок** (`/error-logs`, `/error-logs/stats`). Дані для них
генеруються процедурно при кожному старті контейнера (`backend/data/fake_analytics.php`), а не
беруться з CSV — там від природи великий випадковий time-series набір. Ці сторінки написані
не через `list-framework` (перенесені як є з продакшн-проекту AllSTO), тому як зразок
для розділу 4 орієнтуйтеся на `StoRegistry.vue`, а не на них. У детальній картці візиту
в Аналітиці є вкладка "Діагностика" (ping/traceroute/reverse-DNS/blacklist-check/HTTP-заголовки/
бан IP) — це **справжні** мережеві операції (`backend/src/Admin/Controller/AdminNetworkToolsController.php`),
а не заглушки; при перенесенні в інший проект їх варто або лишити як є (адмін-only,
IP валідується перед підстановкою в shell), або взагалі не переносити, якщо такий функціонал не потрібен.

### Важливо про live-reload

На деяких конфігураціях Windows + Docker Desktop (особливо не системний диск) file-watcher
Vite не бачить правок файлів на льоту. Якщо після правки коду браузер не оновився — зробіть
`docker compose restart frontend` (для фронта) або `docker compose restart backend` (для бека,
PHP перечитується на кожен запит, але правки `composer.json`/додавання нового файлу в новий
неймспейс іноді теж потребують рестарту).

### Продакшн-варіант

`docker-compose.prod.yml` — єдиний контейнер (Apache+PHP), який віддає і зібраний
статичний фронтенд (`npm run build`), і API з одного й того ж процесу. Саме так
зараз розгорнуто на `admin.4n.com.ua`. Дивіться на нього як на приклад деплою за зовнішнім
reverse-proxy (у проді — хостовий Apache з Let's Encrypt).

## 2. Як інтегрувати цю адмінку в інший проект

Основна ідея проекту: більша частина коду адмінки — **не про конкретний продукт**, а про
перевикористовувані механізми. У будь-який новий проект переносяться саме вони, а бізнес-специфіка
(які сутності, які поля) описується конфігом.

### Що перевикористовується цілком

| Механізм | Де лежить | Що робить |
|---|---|---|
| Універсальне вікно деталей | `frontend/src/components/BaseModal.vue` + `frontend/src/composables/useModalWindow.js` | Три режими (floating / docked-right / docked-bottom), drag & resize, автозбереження режиму в localStorage. Контент — через slots (`#title`, default, `#footer`) |
| Каркас списку (список+фільтр+пейджинг+масові операції) | `frontend/src/list-framework/` | `DataListPage.vue` + реєстри типів (`filterTypes.js`, `cellTypes.js`) + `useRemoteOptions.js`. Колонки та фільтри задаються JSON-конфігом, а не хардкодяться в шаблоні. Вибір кількості рядків на сторінці та масові операції (чекбокси, bulk-редагування/видалення) — теж без додаткового конфігу, див. 4.2 |
| Зсув контенту при відкритому докері | `frontend/src/components/ListPageWrapper.vue` + `usePageLayout.js` | Докнуте вікно не перекриває список, а зсуває його |
| Синхронізація фільтрів/сортування/відкритої картки з URL | `frontend/src/composables/useUrlFilters.js` | Фільтри, сортування (зокрема мульти-колоночне) і відкрита детальна картка (id + активна вкладка) переживають перезавантаження сторінки і передаються в посиланні — див. 4.2 |
| Меню, кероване JSON | `frontend/src/config/menu.json` | Пункти меню + права доступу + (задел) ролі — без правки Vue-компонентів |
| Авторизація з dev-заглушкою | `frontend/src/composables/useAuth.js` | `VITE_MOCK_AUTH=true` — пускає з будь-якими даними, поки немає справжнього бекенда; `=false` — ходить у реальний `/api/admin/auth/*` |

### Кроки інтеграції в існуючий проект

1. **Скопіюйте** у цільовий Vue 3 + Vite проект:
   - `frontend/src/list-framework/`
   - `frontend/src/components/BaseModal.vue`, `Pagination.vue`, `SortIcon.vue`, `ListPageWrapper.vue`
   - `frontend/src/composables/useModalWindow.js`, `usePageLayout.js`, `useUrlFilters.js`, `useAuth.js`
   - `frontend/src/config/menu.json` (як зразок структури, наповніть своїми пунктами)
2. **Підключіть свій backend.** `DataListPage.vue` очікує від `apiList` відповідь вигляду
   `{ status, data: [...], pagination: { page, per_page, total, total_pages } }`,
   а `apiUpdate` — `PATCH /{id}` з тілом `{ поле: значення }`. Якщо у вашого API інший
   формат — або адаптуйте його під цей контракт, або поправте `load()`/`handleCellUpdate()`
   у `DataListPage.vue` під свій. Якщо хочете, щоб посилання на відкриту детальну картку
   (`?id=...`) працювало при прямому переході/перезавантаженні (а не лише при кліку по рядку
   вже завантаженого списку) — додайте також `GET {apiUpdate}/{id}` (один об'єкт), див.
   `AdminStoController::show()` і `StoRegistry.vue`'s `useUrlFilters({ detail: { id, onOpen } })`
   як зразок.
3. **Опишіть свою сутність** трьома JSON-файлами за зразком
   `frontend/src/pages/sto-registry.{filter,columns,config}.json` (детальніше — розділ 4).
4. **Стилі.** Зараз використовується Bootstrap 5 — якщо в цільовому проекті інший UI-кит,
   компоненти `list-framework/filters/*` і `list-framework/cells/*` достатньо прості,
   щоб переписати розмітку під вашу дизайн-систему, не чіпаючи логіку в `DataListPage.vue`.

### Backend — за бажанням

Бекенд на Yii3 у цьому проекті — **демонстраційний** (фейкові дані, SQLite замість реальної
БД). Для інтеграції в реальний проект backend зазвичай береться не звідси, а пишеться під
реальну БД вашого проекту — важливо лише повторити контракт відповідей (див. вище) і, якщо
потрібен Swagger, — стиль анотацій `#[OA\...]` з `backend/src/Admin/Controller/*`
(атрибути пакета `zircote/swagger-php`).

## 3. Інтеграція за допомогою AI (Claude)

Цей проект спеціально спроєктований так, щоб інтеграцію робив не тільки людина руками,
а в першу чергу — AI-асистент (наприклад, [Claude Code](https://claude.com/claude-code)),
якому дали доступ і до цього репозиторію, і до цільового проекту. Вся механіка винесена
в зрозумілі, задокументовані, невеликі файли (JSON-конфіги, реєстри типів) саме для того,
щоб її було легко "прочитати" і повторити за аналогією — це рівно те, у чому AI-асистенти
особливо хороші.

Практичний сценарій:

1. Відкрийте Claude Code (або аналог) у цільовому проекті, до якого потрібно підключити адмінку.
2. Дайте йому доступ так само і до цього репозиторію (наприклад, клонуйте поряд).
3. Попросіть приблизно так:

   > Вивчи `frontend/src/list-framework` і `frontend/src/components/BaseModal.vue`
   > у проекті `admin` — це каркас списку з фільтром/пейджингом і універсального вікна
   > деталей. Перенеси ці механізми в мій проект і підключи їх для сутності `<назва>`:
   > колонки `<...>`, фільтри `<...>`, детальна картка — докер справа. Дивись на
   > `frontend/src/pages/StoRegistry.vue` і `sto-registry.*.json` як на зразок, як
   > це поєднується на одній сторінці.

4. Для нестандартних елементів (свій тип фільтра, своя комірка таблиці) — попросіть
   зареєструвати новий тип через `registerFilterType()`/`registerCellType()`
   (див. розділ 4) замість того, щоб писати список руками з нуля.

AI добре справляється саме з цим видом роботи: контракт (JSON-схема, форма відповіді API)
чіткий і невеликий, а типових сторінок зазвичай багато і вони однотипні — тобто після
однієї показаної вручну інтеграції інші сутності AI може додавати за аналогією
майже самостійно.

## 4. Як додавати нові сторінки та елементи

### 4.1. Проста сторінка (без списку)

1. Створіть `frontend/src/pages/MyPage.vue` (звичайний Vue-компонент, оберніть у `BaseLayout`
   або `ListPageWrapper`, якщо плануєте докер-вікно).
2. Додайте маршрут у `frontend/src/router/index.js`:
   ```js
   { path: '/my-page', name: 'MyPage', component: () => import('../pages/MyPage.vue') },
   ```
3. Додайте пункт у `frontend/src/config/menu.json` (в існуючу групу або нову):
   ```json
   { "label": "Моя сторінка", "to": "/my-page", "icon": "bi-star", "permission": "my.view", "roles": ["admin", "superadmin"] }
   ```

### 4.2. Список із фільтром і пейджингом (типовий випадок)

Візьміть за зразок `frontend/src/pages/sto-registry.*.json` і `StoRegistry.vue`.

1. **Колонки** — `my-entity.columns.json`, масив об'єктів:
   ```json
   { "key": "name", "label": "Назва", "type": "text", "sortable": true, "editable": true }
   ```
   `type` — береться з реєстру `frontend/src/list-framework/cellTypes.js`
   (`text`, `select`, `boolean`, `number`, `phone-list` — останній зберігає масив рядків і
   форматує їх під "+38 (0XX) XXX-XX-XX", див. `cells/PhoneListCell.vue` і `utils/phone.js`
   як готовий зразок кастомного типу зі своїм форматуванням і E.164-нормалізацією при
   збереженні). `editable: true` вмикає inline-редагування прямо в таблиці (без
   `editable` — лише читання).

   `sortable: true` вмикає клік по заголовку колонки для сортування. Звичайний клік —
   сортування лише за цією колонкою; **Ctrl/Cmd+клік** — додає колонку до вже вибраного
   сортування (мульти-колоночне сортування, порядок кліків = пріоритет). Стан зберігається
   в URL одним параметром `?sort=key:ASC,key2:DESC` (через `multiSort` у `useUrlFilters`),
   на бекенд передається як `sort_by=key,key2&sort_dir=ASC,DESC` (список полів/напрямків
   у тому ж порядку). **Важливий нюанс:** якщо поле в БД зберігає код/id, а користувачу
   показується підпис (напр. `select`-колонка з `options`) — сортувати за сирим кодом
   неправильно, треба сортувати за словом, яке видно в таблиці. На бекенді для такого
   поля підміняйте `ORDER BY` на `CASE ... WHEN 'code' THEN 'Label' ... END` — приклад:
   `AdminStoController::sortExpr()` для колонки `sto_type`.

2. **Фільтр** — `my-entity.filter.json`, аналогічно, типи — з `filterTypes.js`
   (`text`, `select`, `checkbox`). Для випадаючого списку, що наповнюється з бекенда:
   ```json
   { "key": "country_id", "type": "select", "label": "Країна", "optionsUrl": "/api/admin/geography/countries", "optionsValueKey": "id", "optionsLabelKey": "name_uk" }
   ```
   Опції кешуються за URL — один і той самий довідник не завантажується повторно, навіть якщо
   використовується в кількох полях/сторінках.

3. **Конфіг ендпоінтів** — `my-entity.config.json`: `apiList`/`apiUpdate`/`apiDelete`,
   дозволи, кнопки дій у рядку (`detail`, `delete`, кастомні).

4. **Пейджинг і масові операції працюють без додаткового конфігу** — вони не описуються
   в JSON, `DataListPage.vue` вмикає їх завжди:
   - **Кількість рядків на сторінці** — випадаючий список (5 / 10 / 20 / 50 / 100 / 250,
     за замовчуванням 50) справа від пейджера. Як і сортування, синхронізується з URL
     (`?per_page=100`) через той самий `useUrlFilters`. Бекенд повинен вміти приймати
     `per_page` в `GET {apiList}` — див. `AdminStoController::list()` (там же максимум,
     до якого обмежується значення, — під свій бекенд можна зменшити/збільшити).
   - **Масові операції** — чекбокс перед кожним рядком (і «вибрати всі на сторінці»
     в заголовку). Щойно вибрано хоч один рядок, над таблицею з'являється панель: випадаючий
     список полів — це рівно ті колонки, у яких у конфігу стоїть `"editable": true`
     (нічого окремо перелічувати не потрібно), а під ним — **той самий** компонент комірки,
     що й для inline-редагування цього поля в таблиці (той самий `cellTypes`-реєстр, той самий
     контракт `field`/`modelValue`/`readonly`/`row`+`update:modelValue`) — новий тип UI під
     bulk-редагування писати не довелося. «Застосувати» проганяє `PATCH {apiUpdate}/{id}`
     по черзі (не паралельно — SQLite-подібні бекенди погано переносять пачку одночасних
     записів) для кожного вибраного id. Кнопка «Видалити» з'являється лише тоді, якщо в `actions`
     конфігу сутності є дія `type: "delete"` і в користувача є її `permission`
     (використовується та сама перевірка, що й для кнопки видалення в рядку) — окремо налаштовувати
     її для bulk-панелі не потрібно. Вибір завжди обмежений поточною завантаженою сторінкою і
     скидається при зміні сторінки/фільтра/сортування/розміру сторінки.

5. **Сторінка** — тонкий `.vue`-файл, який просто передає ці три JSON у `DataListPage`
   (див. `StoRegistry.vue` цілком — там же приклад відкриття детальної картки в `BaseModal`
   по кліку на кнопку `detail`). Там же — зразок, як зробити так, щоб посилання на відкриту
   картку (і, якщо в картки кілька вкладок — на конкретну вкладку) переживало
   перезавантаження і годилося для передачі іншій людині: свій `useUrlFilters({ filters: { tab: activeTab }, detail: { id: detailId, onOpen } })`
   на рівні сторінки (окремо від того, що `DataListPage` вже робить для фільтрів списку) —
   `onOpen(id)` підвантажує запис через `GET {apiUpdate}/{id}` і відкриває `BaseModal`,
   `activeTab` синхронізується як звичайний фільтр. У `sto-registry.config.json` кнопки
   дій (`actions`) теж можуть відкривати одразу конкретну вкладку через `"tab": "..."`.

### 4.3. Власний (кастомний) елемент фільтра або колонки

Якщо бібліотечних типів (`text`/`select`/`boolean`/`number`/`checkbox`) не вистачає:

```js
// у вашому коді ініціалізації (наприклад, main.js) або прямо в компоненті сторінки
import { registerCellType } from '@/list-framework/cellTypes'
import MyRatingCell from '@/components/MyRatingCell.vue'

registerCellType('my-rating', MyRatingCell)
```

Компонент повинен приймати props `field`, `modelValue`, `readonly`, `row` і емітити
`update:modelValue` — той самий контракт, що й у вбудованих `TextCell.vue`/`SelectCell.vue`
(дивіться будь-який з них як зразок). Після реєстрації просто вкажіть `"type": "my-rating"`
у JSON-конфігу колонки.

### 4.4. Детальна картка (вікно)

Будь-яка кількість вкладок/полів — звичайна розмітка всередині `<BaseModal>`. Три режими
відображення, drag/resize, збереження розмірів — вже працюють без додаткового коду.
Дивіться `frontend/docs/BASE_MODAL_USAGE.md` і `frontend/docs/MODAL_WINDOWS.md` — там детальний опис
усіх props і слотів з прикладами.
